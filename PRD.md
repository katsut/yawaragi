## 1. 概要
テキストコミュニケーションで受け取ったキツい一言から**受け手を守る**、オンデバイスの感情バッファ。Slack web / Backlog コメントに 🛡️ ボタンを注入。押すと、その場のスレッド文脈を読み、**①温度 ②善意の再解釈 ③落ちてる文脈 ④冷静な返信ドラフト**を返す。**処理は端末内で完結し、何も外部送信しない。**

## 2. 背景 / Why Now
AIが下書き・要約・代返・エージェント間対話を担い、**人同士の直接対話は減っていく**。残る接点はほぼテキストで、トーン・文脈・背景が落ちるため、言葉は実際よりキツく届く。交流が薄いほど"関係の貯金"が少なく、誤読と感情事故のダメージが増す。
→ **対話が痩せる時代に、関係構築はより難しくなる。** Yawaragi は、その痩せた人間チャンネルを事故から守るインフラ。対話とコラボを助けるプロダクト（Backlog等）と同じ使命線上にある。

## 3. 課題
- テキストは表現が一チャンネル（文字）のみ。トーン/文脈/背景が欠落 → 誤読・"キツさ"の増幅
- 刺さり具合は受け手のその日の**心のキャパ**に依存する
- 反射的な怒り返信が関係を壊す
- 職場(Slack/Backlog)は非匿名 →「誰が言ったか（序列）」が刺さり方を増幅する

## 4. ターゲットユーザー
- **主:** Slack/Backlog で日々テキストを受け取る働き手（受信側の個人）
- **チャンピオン=本人**。自分で入れて自分で使う（購買判断・IT承認が不要な個人ツール）

## 5. 提供価値
キツい一言に傷ついた瞬間、**押すと**——落ちていた文脈を補い、相手を寛容かつ現実的に読み直し、後悔しない返信を用意する。**怒り返信を思いとどまり、関係が生き延びる。** そして**何も端末外に出ない**（＝IT承認不要、厳しい社内でも使える）。

## 6. ゴール / ノンゴール
**ゴール**
- 誤読起因の衝突を減らす／de-escalation／関係の保全
- プライバシー純度（外部送信ゼロ）／厳しい社内環境でも使える
- Chrome拡張として **公開可能な本物**（デモ用ハックでない）

**ノンゴール（今回）**
- 送信側のトーンチェック（v2）
- チーム/組織の分析ダッシュボード（＝従業員の感情監視。倫理的に採らない）
- クラウド処理（既定）／匿名化／人間の判断の代替／自動スキャン

## 7. 設計原則（North Star）
1. **オンデバイス完結**。何も外部送信しない。バックエンドなし
2. **感情を否定しない**（validate, not invalidate）。toxic positivity 禁止
3. **人から遠ざけない**。相手を理解し直し、関係に"より良い状態で戻す"道具
4. **受信側・手動トリガーのみ**

## 8. スコープ（MVP）
- 対象: **Slack web / Backlog コメント**
- 器: **Chrome拡張（MV3・純クライアント）**
- AI: **オンデバイス Gemini Nano**（Prompt API、拡張向けstable / Chrome 138+）
- トリガー: 受信側・手動 🛡️
- 出力: 温度 / 善意の再解釈 / 落ちてる文脈 / 冷静な返信ドラフト
- 文脈: **Tier0＝画面上のスレッド**（DOMから読む）＋ **Tier1＝今日のキャパ**（ワンタップ）
- 返信ドラフトのコピー / キャパのローカル保存

## 9. スコープ外（MVP）
送信側 / ネイティブSlackアプリ / バックエンド / 分析 / 経時学習・相手別プロファイル / クロスデバイス / StromaDB / クラウド（dev fallback以外）/ 自動スキャン

## 10. 機能要件（FR）
- **FR1** Slack web と Backlog のメッセージ/コメントに、ホバーで 🛡️ を注入する
- **FR2** 押下で、対象本文・直近スレッド(〜5件)・送信者名を画面DOMから取得する（外部送信しない）
- **FR3** 取得内容＋今日のキャパをオンデバイスLLMに渡し、4要素を得る
- **FR4** 結果を Shadow DOM の popover に4セクションで表示する
- **FR5** 「冷静な返信ドラフト」をワンクリックでコピーできる
- **FR6** 今日のキャパ(低/普通/高)を切替・ローカル保存し、生成に反映する
- **FR7** モデル利用不可時は丁寧に案内し、設定ON時のみクラウドfallbackへ（既定OFF・キーは持たない）
- **FR8** fallback OFF時、いかなるデータも端末外に送信しない

## 11. 非機能要件（NFR）
- **プライバシー:** 既定で外部送信ゼロ。権限は `storage` と slack/backlog の host のみ。リモートコードなし
- **性能:** 体感で待たされない（目標: 数秒以内）
- **互換:** MV3 / Chrome Web Store 公開可能
- **言語:** 日本語UI・日本語出力品質を重視
- **倫理:** 出力は原則(感情を否定しない)に準拠

## 12. UX（1画面 popover）
🛡️ → 温度(体感/実際) → 善意の再解釈 → 落ちてる文脈 → 冷静な返信ドラフト[コピー] → 🔒 端末内で処理・送信なし ／ 今日のキャパ[低][普通][高]

## 13. 成功指標
- **ハッカソン:** 実在の Slack/Backlog メッセージで、文脈を踏まえた再解釈＋冷静返信が**端末内だけ**で出る動作デモ。審査員に "why（対話が痩せる時代の関係保護）" が伝わる
- **プロダクト(将来):** 「怒り返信を思いとどまった」体験率 / 週次継続 / インストール継続

## 14. 制約・依存
- 利用側: Chrome 138+ / 対応OS(Win10-11・macOS13+・Linux・ChromeOS) / 一定のハード要件 / 初回 ~2GB モデルDL
- DOMセレクタはサービス依存で脆い（要ライブ調整）
- 受信側のみ

## 15. リスク
- Nano の日本語品質 → 前夜チェック済 / タスクを狭く / dev fallback 保険
- DOM変更耐性 → 汎用化せずデモの1パターンを死守
- **過剰仲介**（North Star違反：人を遠ざける"感情のコンドーム"化）→ 出力を"関係に戻す"設計で回避
- 利用端末でNano非対応 → availability検知＋グレースフル案内（公開版の必須要件）

## 16. 将来（v2+）

- ローカルの「関係文脈グラフ」（将来 StromaDB を**ローカルで**検討＝プライバシー維持のまま経時文脈）
- クロスチャネル文脈
- ガバナンス思想「中身は匿名評価 / 責任は記名」
