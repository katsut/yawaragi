## 依存度レーン（クリティカルパスは高依存レーンだけ）

**🔴 高依存レーン（外部環境・実機・人間の時間が必要 ＝ 真のクリティカルパス）**
```
S1(実機確認5分) → S2 Nano品質検証(30〜60分 ★最大のボトルネック)
                → S3 実DOMライブ調整(〜30分×2・ユーザー同席)
                → S6 通しリハ(30分)
```

**🟢 低依存レーン（純コード。AI がバックグラウンドで並行完走・障害にならない）**
```
S1 スキャフォールド / S4 Popover UI(モックJSONで先行) / S5 キャパtoggle
S7 呼吸ガイド / S8 関係性トグル / LP(lp/index.html・拡張と独立)
```

- S1 完了後、**低依存レーンは全部並行で subagent に委譲**できる（ファイル分割: ai.js / popover.js / storage.js / breath.js）
- スケジュールを決めるのは高依存レーンの4点だけ:
  ① S1 load unpacked 確認（5分）② S2 Nano 品質検証（30〜60分）③ S3 ライブ調整（〜30分×2サービス）④ S6 リハ（30分）
- **ウォールクロック目標: 半日**（支配項はコード生成ではなく実機検証）

## S1. 拡張スキャフォールド＋🛡️注入 — AI実装 15〜30分 ／ 実機確認 5分
manifest（MV3・最小権限）と content script の骨組みを作り、対象ドメインで 🛡️ が出るところまで。

**受け入れ条件**
- [ ] `manifest.json` が ARCHITECTURE §3.1 と一致（permissions は `storage` のみ、host は slack/backlog のみ、リモートコードなし）
- [ ] `chrome://extensions` で load unpacked が警告なしで通る
- [ ] Slack web / Backlog のメッセージ hover で 🛡️ ボタンが表示される（クリックはまだ console.log で可）
- [ ] content script ↔ service worker のメッセージパッシング配線が疎通している（Prompt API は content script から直接呼べない可能性が高いため、S2 の前に配線を済ませる）

## S2. AI module（Nano 接続＋プロンプト＋パース）— AI実装 30分 ／ 実機検証 30〜60分 ★最大リスク
`LanguageModel.availability()` → session 作成 → ARCHITECTURE §3.5 の system prompt で4要素 JSON を得る。

**受け入れ条件**
- [ ] availability 判定と「利用不可時の丁寧な案内」がある（FR7）
- [ ] §3.5 の system prompt を使い、`{temperature{felt,actual}, reinterpretation, missing_context, calm_reply}` が返る（temperature 先頭固定）
- [ ] `responseConstraint`（JSON schema）を試し、不可なら safeParse＋素朴パースの二段（§8）
- [ ] キャパ(低/普通/高)がプロンプトに反映される（FR6）
- [ ] 実行コンテキスト（content script / offscreen / service worker）を公式サンプルで確認し、動いた構成を ARCHITECTURE に追記
- [ ] **テストマトリクス5ケース**（placeholder 名で自作）で安全ガードの発火を検証:
  1. 曖昧・そっけない低強度（「了解です。」等）→ 通常の4出力
  2. 句点/短文の誤読系 → missing_context が文体の可能性に触れる
  3. 業務範囲の明確な批判（中強度）→ 再解釈は出るが感情の受け止めが先
  4. 人格攻撃級（高強度）→ **再解釈しない**・距離を置く返信になる
  5. キャパ「低」×中強度 → 受け止めが厚く・再解釈が軽くなる
- [ ] **実機検証タイムボックス60分**: 品質が出なければ §3.5 の削減優先順位でプロンプト短縮（AI側の再生成は数分）→ それでも不可なら dev fallback で先へ進む

## S3. DOM 抽出（Slack / Backlog 各1パターン）— AI実装 15分 ／ ライブ調整 〜30分×2（ユーザー同席）
クリックした対象の本文・直近スレッド（〜5件）・送信者名を DOM から取る。汎用化しない。
セレクタは実画面の inspect 結果が正。AI が推測で書いたセレクタは動かない前提で、実画面での調整時間を確保する。

**受け入れ条件**
- [ ] Slack web: デモで使うチャンネル/スレッド画面で `{targetText, thread, sender}` が取れる
- [ ] Backlog: デモで使う課題コメント画面で同上
- [ ] 外部送信ゼロ（fetch 等が存在しない。FR8）

## S4. Popover UI（shadow DOM・4出力＋コピー＋認知安全弁）— AI実装 30〜45分 ／ 実機確認 10分
§3.6 のレイアウト。CSS 衝突回避のため shadow DOM。**モック JSON で S2 と並行開発**。

**受け入れ条件**
- [ ] **即時受け止め表示（P0）**: popover は押下と同時に開き、生成を待たず固定の受け止め文を先行表示 → temperature 到着で差し替え（スピナーだけを見せない）
- [ ] 表示順固定: 温度（受け止めが先）→ 再解釈 → 文脈 → 返信ドラフト（validate→reappraise）
- [ ] [コピー] で calm_reply がクリップボードに入る（FR5）
- [ ] 「読み直しを畳む」で再解釈/文脈が collapse し受け止めだけ残る（認知安全弁・P1）
- [ ] 🔒「端末内で処理・送信なし」表示がある

## S5. キャパ toggle＋ローカル状態 — AI実装 10分（S2/S4 と並行可）
**受け入れ条件**
- [ ] [低][普通][高] の切替が `chrome.storage.local` に保存され、次回生成に反映（FR6）
- [ ] dev 用 fallback トグル（デフォルト OFF・本番導線に出さない）（FR7）

## S6. 仕上げ＋デモリハーサル — 人間中心 30〜60分（AI はバグ修正の即応担当）
**受け入れ条件**
- [ ] デモ台本（ARCHITECTURE §7）どおり Slack→Backlog の流れが通しで動く
- [ ] テストデータは placeholder 名（Alice/Bob 等）のみ。実在の同僚メッセージ不使用
- [ ] ネットワークタブで外部送信ゼロを実演できる

## ストレッチ（低依存・AIがバックグラウンドで完走可能。高依存レーンを一切妨げない）

### S7. 90秒の一拍 — 呼吸ガイド UI — AI実装 20分 ／ 🟢低依存
高強度判定時 or キャパ「低」のとき、popover 冒頭に「まず一拍」ミニガイド（循環ため息×3呼吸の CSS アニメーション）を差し込む（researches ②: 生理的クールダウンが再解釈の前提条件）。

**受け入れ条件**
- [ ] 表示条件は「高強度 or キャパ低」のときのみ（毎回は出さない・押し付けない）
- [ ] ワンタップでスキップ可能
- [ ] CSS アニメーションのみ（外部リソースなし）

### S8. 関係性トグル（相手: 上司/同僚）— AI実装 15分 ／ 🟢低依存
キャパの隣に任意トグル。選択時のみプロンプトに序列文脈を追加（researches ①: 部下は上司からの曖昧メッセージを構造的に過剰ネガティブ解釈）。

**受け入れ条件**
- [ ] デフォルト未選択。未選択時は現行動作と完全に同一
- [ ] 選択は `chrome.storage.local` に保存（メッセージ単位でなくセッション単位）

### LP（ランディングページ）— 🟢低依存・拡張コードと独立
`lp/index.html`（静的1ファイル・外部送信ゼロの思想をページ自体でも体現）。プランニング完了後に作成。
